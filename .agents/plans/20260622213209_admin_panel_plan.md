# Admin panel implementation plan

## Context

Zeta needs an operational area inside the existing Angular dashboard. The MVP
must let administrators inspect users, every uploaded video, and every coaching
session; change user roles; manage a separate kind of admin-issued expert code;
and inspect admin activity. It is not a content moderation system yet.

Identity and roles remain owned by WorkOS, product data by PostgreSQL/sqlc, and
video media by Mux. The existing Go API remains the only backend. Unfold/Django
is not part of this implementation because it would add a second runtime,
authentication boundary, deployment, and data-access path.

## MVP scope

Build an `/admin` area in `web/dashboard-next`, backed by explicit `/admin/*`
endpoints in the Go API.

Included:

- paginated Users list with role changes;
- paginated Videos list across all groups and owners, with read-only playback;
- paginated Sessions list across all groups, experts, and students;
- server-side filters with searchable user/group selectors;
- admin-issued expert codes, separate from personal referral codes;
- read-only admin Activity list based on the existing audit trail.

Explicitly excluded:

- permanent user deletion or staged erasure (separate ticket);
- deactivate/reactivate;
- moderation statuses, restrict/restore, or Mux deletion;
- last-admin protection if it requires cross-system reconciliation. Cheap
  self-role-change protection is still desirable to avoid an accidental lockout.

## Authorization

Use granular permissions rather than `role == admin` checks:

- `admin:users:read`
- `admin:users:role:update`
- `admin:videos:read`
- `admin:sessions:read`
- `admin:codes:read`
- `admin:codes:manage`
- `admin:audit:read`

Add them to `internal/permissions/permissions.go` and the Angular permission
union. Create them through the WorkOS Authorization API, add them to the admin
role additively, and verify assignments in each target environment. Angular
guards and hidden navigation improve UX; every Go route must independently
enforce its permission.

## API conventions

Use server-side pagination for every table:

```json
{
  "items": [],
  "page": 1,
  "page_size": 25,
  "total": 0,
  "total_pages": 0
}
```

- Accept `page` and `page_size`; default to 1/25 and cap page size at 100.
- Use a stable default order (`created_at DESC, id DESC`, or equivalent).
- Return an empty page, not an error, when filters have no matches.
- Keep filters in URL query parameters so back/forward, refresh, and shared
  internal links preserve the current view.
- Reset to page 1 whenever a filter changes.
- Implement count and page queries together in `db/queries/admin.sql`; run
  `make db:sqlc` after query changes.

## Backend

### Foundation

1. Add `internal/admin` handlers/services and mount `/admin` inside the existing
   authenticated, active-access router group.
2. Add shared parsing for pagination, dates, and bounded filter text.
3. Add focused WorkOS user-resolution helpers so page queries stay batched and
   provider failures are surfaced without leaking raw PII to logs.

### Users

1. `GET /admin/users` lists the local Zeta population from `user_access`, joined
   with preferences where available, and enriches only the returned page with
   WorkOS user/membership data.
2. Filters: name or exact user ID, role, and access status. Do not duplicate
   WorkOS email into a new local column just to support fuzzy email search.
3. `PUT /admin/users/{userID}/role` allowlists `student|expert|admin`, updates the
   default-organization membership in WorkOS, and revokes the target user's
   sessions so stale JWT permissions do not remain usable.
4. Reject changing one's own role if this is straightforward in the handler.
   Do not delay the MVP for last-active-admin reconciliation.
5. Record a curated `user.role_updated` audit event. Since WorkOS and PostgreSQL
   cannot share a transaction, report provider/audit partial failures explicitly
   rather than claiming atomicity.

### Videos

1. `GET /admin/videos` lists local `assets` with their video parts, owner, group,
   group owner, relevant expert identities, processing state, duration, creation
   time, Mux IDs, and playback IDs.
2. Filters: owner, expert, group, asset/video status, and created date range.
   User filters accept stable user IDs; labels are resolved names with the ID as
   secondary text.
3. The endpoint bypasses normal group visibility only because it requires
   `admin:videos:read`. No existing public/user asset query is broadened.
4. Use persisted Mux data for the list. Avoid one Mux request per row; fetch Mux
   only when required data is genuinely absent from the local record.

### Sessions

1. `GET /admin/sessions` lists all coaching bookings with group, session type,
   expert, student, scheduled time, duration, cancellation/status, recording
   status, and linked recording asset where available.
2. Filters: expert, student, group, session status, recording status, and
   scheduled date range.
3. Reuse the current booking response/presentation rules where practical, but
   keep the admin query separately permissioned and paginated.

### Filter options

Add small paginated lookup endpoints for remote comboboxes rather than loading
all options into the browser:

- `GET /admin/options/users?q=&role=&page=` searches local display name or exact
  WorkOS user ID and returns `{value, label, description}`;
- `GET /admin/options/groups?q=&page=` searches group name or exact group UUID.

The options API is scoped by the matching admin read permission. Debounced
search should require a short minimum input except when showing a small initial
recent set.

### Expert codes

Current `signup_codes` are five automatically maintained personal referral
codes per expert; manual creation/revocation was intentionally removed. Preserve
that invariant.

1. Add a separate admin-issued code table with purpose, status, optional expiry
   and max uses, creator, redemption count, and timestamps.
2. Add `GET/POST /admin/codes` and `POST /admin/codes/{id}/revoke`.
3. Extend redemption transactionally to accept either personal or admin-issued
   codes. Show plaintext only on creation and never write bearer codes to logs
   or audit snapshots; prefer storing a keyed hash where practical.
4. Audit create/revoke/redemption lifecycle changes with curated snapshots.

### Activity

Add `GET /admin/activity` over the append-only audit table, filtered to admin
actions initially. Paginate it and support actor, action, resource type, and
date range filters. Do not return sensitive metadata or uncurated snapshots.

## Frontend

### Information architecture

1. Add one Admin sidebar item visible when any admin read permission is present.
2. Use an `/admin/:section` shell with tabs or compact local navigation for
   Users, Videos, Sessions, Codes, and Activity.
3. Each route is independently guarded by its matching permission; inaccessible
   sections are omitted rather than shown disabled.

### Tables and responsive behavior

1. Build dense operational tables with a small set of primary columns and
   secondary IDs beneath names. Avoid horizontally displaying every provider
   field.
2. On narrow screens, switch rows to compact stacked records rather than forcing
   an unusable full-width table.
3. Use skeleton rows while loading and explicit empty/error states. Preserve the
   current table while a new page/filter request is in flight when possible.
4. Put row count and `25/50/100` page-size selection beside pagination.

### Pagination

Create a reusable `z-pagination` wrapper around the installed
`ng-primitives/pagination` API (`NgpPagination`, previous/next, page buttons).
It receives `page`, `pageCount`, `total`, and loading/disabled state, emits the
new page, uses lucide icons, includes accessible labels, and collapses the page
window with ellipses. Add focused tests and a Storybook story.

### Searchable filters

Extend the existing `z-combobox` instead of introducing a second combobox. Its
current positioned-dropdown directive fixes first-pass width and overlay
visibility and must be preserved.

- Add an optional input/search mode using `NgpComboboxInput` while leaving the
  current select-like mode unchanged for existing callers.
- Support externally supplied remote options, loading/no-results text, clear,
  and a search-change output; debounce/cancel HTTP requests in the page/store.
- Render display name as the label and user ID as secondary text; selecting an
  option stores the ID, never the mutable label.
- Verify keyboard navigation, focus return, escape/outside click, portal width,
  empty results, and rapid-query race handling against the existing regression
  tests.

### Video preview

Use a read-only dialog for the MVP so opening a video does not lose the table's
page and filters. Reuse the existing Mux player setup and shared dialog patterns;
show asset title, owner/group context, and allow switching between video parts.
Do not reuse the normal asset endpoint because its group visibility rules must
remain intact. A dedicated admin detail page can be added later if review or
diagnostic workflows outgrow the dialog.

### Role and code actions

- Role changes use the existing confirm dialog, clearly showing user and old/new
  role, with disabled/submitting/error states.
- Code creation uses a form dialog; revoke uses a destructive confirmation.
- All user-facing text goes through Transloco.

## Main areas touched

- `internal/admin`, `internal/api/server.go`, `internal/permissions`
- WorkOS client abstraction/mocks and audit resource/action constants
- `db/queries/admin.sql`, access queries, reversible code-table migrations, and
  generated `internal/db` code
- `web/dashboard-next/src/app/app.routes.ts`, shell navigation, admin pages,
  API clients/stores, and Transloco locales
- `shared/ui/combobox`, new `shared/ui/pagination`, dialogs, skeletons, and video
  preview/player composition
- `README.md` admin access and expert-code flows plus architecture diagrams

## Delivery order

1. Permissions, admin routing, pagination contract, and filter-option endpoints.
2. Read-only Videos and Sessions pages with filters, pagination, and video dialog.
3. Users list and role changes.
4. Admin-issued expert codes and redemption integration.
5. Activity page and final audit/UX polish.

This order delivers the highest-value inspection tooling first and keeps each
increment independently testable.

## Verification

- Backend tests for every permission, page boundary, filter combination, stable
  ordering, empty result, provider failure, and role update/audit path.
- Integration tests for admin list queries and admin-issued code redemption;
  run `make db:sqlc` after SQL changes.
- Angular tests for route/nav permissions, URL-synchronized filters, stale
  request cancellation, responsive rows, dialog playback, role/code actions,
  searchable combobox regressions, and pagination keyboard/disabled behavior.
- Run `make test:unit`, relevant `make test:integration`, `make api:build`,
  `make web-next:lint`, `make web-next:test`, and `make web-next:build`.
- Run `make web-next:storybook:build` because a shared pagination component and
  searchable combobox behavior are being added.

## Deferred tickets

- permanent user deletion and retention policy;
- deactivate/reactivate;
- content moderation state and media removal;
- last-active-admin reconciliation if not trivial during role implementation.
